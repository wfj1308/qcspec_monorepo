# 上线验收工程标准

## 1. 目标
本文档定义上线级交付的工程与验收标准。
适用于前端与后端开发，且对新功能、重构、缺陷修复均为强制要求。

目标：
- 通过模块化与组件化设计提升可维护性。
- 通过测试与发布门禁保障可靠性。
- 通过清晰文档与验收记录保障可追溯性。

## 2. 适用范围
- 前端：`apps/web/**`
- 后端：`services/api/**`
- 共享包与契约：`packages/**`
- 影响运行时行为的基础设施与配置变更。

## 3. 不可妥协的完成定义（DoD）
仅当以下全部成立时，任务才可视为“完成”：
- 代码符合项目规范并通过静态检查。
- 架构保持模块化，不新增巨型“上帝文件/上帝组件”。
- 对行为变更补充或更新单元/集成测试。
- API 或契约变更在需要时完成文档化与版本化。
- 用户可见行为通过验收清单验证。
- 关键路径新增监控/日志/错误处理已完整。
- 不存在已知阻断级缺陷。

## 4. 工程原则（前后端统一）
- 单一职责：一个模块只解决一个明确问题。
- 边界稳定：领域逻辑、编排逻辑、IO/UI 关注点要分离。
- 契约明确：跨模块交互必须有类型化接口。
- 默认向后兼容：破坏性变更必须附迁移说明。
- 失败可见：结构化错误、可执行日志，不允许静默失败。
- 安全优先：代码不写死密钥，不信任未校验输入。

## 5. 前端实现标准

### 5.1 结构与模块化
- 使用组件化 UI 组合，避免超大页面组件。
- 业务逻辑放在 hooks/services，不放在 JSX 渲染块中。
- 文件职责必须聚焦：
- 组件文件：展示与交互编排。
- Hook 文件：状态与行为编排。
- 工具文件：纯函数与可复用辅助逻辑。
- 新增逻辑默认可复用，除非需求明确要求强耦合。

### 5.2 质量规则
- 保持 TypeScript 严格性；除非有注释说明，否则不使用 `any`。
- 避免重复逻辑；共享能力抽到 hooks/utils。
- 避免隐藏副作用；异步流程必须有清晰 loading/error 状态。
- 用户文案必须可读，不允许编码乱码。
- 满足可访问性基线：键盘可达、标签完整、语义化控件。

### 5.3 前端测试与验证
- 必须通过 `npm --workspace @qcspec/web run lint`。
- 必须通过 `npm --workspace @qcspec/web run build`。
- 需为以下场景新增或调整测试：
- 关键流程状态流转。
- 异常态与空态。
- 数据解析/转换逻辑。
- 对本次改动影响的主要用户流程必须完成人工验收。

## 6. 后端实现标准

### 6.1 结构与模块化
- handler/controller 保持轻量，领域逻辑下沉到 service 层。
- service 层聚焦业务，持久化/外部调用/解析逻辑需隔离。
- 纯转换和复用校验放到 helper/util 模块。
- 新增接口不得混写传输层关注点与领域规则。

### 6.2 API 与契约规则
- 请求/响应 schema 必须明确且稳定。
- 输入校验在边界层完成，非法请求尽早拒绝。
- 错误返回结构统一且机器可读。
- 长任务必须提供清晰状态模型（state/progress/message/error）。

### 6.3 可靠性与可观测性
- 关键生命周期和失败场景必须有结构化日志。
- 异步任务及跨服务链路尽量提供关联标识。
- 外部依赖调用必须显式设置超时与重试策略。
- 关键路径禁止吞异常。

### 6.4 后端测试与验证
- 领域逻辑与纯转换必须有单元测试。
- 接口行为与持久化交互必须有集成测试。
- 外部接口在适用时需契约测试。
- 涉及 schema/数据变更时必须有迁移与安全性验证。

## 7. Code Review 门禁（强制）
评审未确认以下全部项前，PR 不得合并：
- 架构：模块边界清晰且未退化。
- 可读性：无需依赖“口口相传”即可理解。
- 正确性：成功路径与失败路径均已实现。
- 安全性：鉴权、校验、数据完整性已处理。
- 可测试性：测试覆盖变更面且有实际价值。
- 可运维性：日志、指标挂点、回滚策略明确。

## 8. 上线验收清单
生产发布前必须按以下清单执行：

### 8.1 功能验收
- 核心场景端到端可用。
- 异常场景反馈符合预期且用户可理解。
- 关联主流程无回归。

### 8.2 技术验收
- 前端 lint/build 通过。
- 后端 lint/test/build 通过。
- 无阻断级或严重级已知缺陷。
- 变更路径性能满足基线预期。

### 8.3 数据与契约验收
- API 契约变更已文档化。
- 向后兼容已验证，或迁移方案已审批。
- 导入/导出/解析行为已用真实样本验证。

### 8.4 运维验收
- 日志与告警足以支持故障排查。
- Runbook 或排障文档已更新。
- 高风险变更需在预发验证回滚方案。

## 9. 重构专项要求
- 重构默认必须保持行为一致，除非变更已明确审批。
- 必须补充重构前后说明：范围、风险点、验证证据。
- 大文件优先按职责拆分，再做内部优化。
- 增量重构期间尽量保持对外接口稳定。

## 10. 文档要求
每个影响生产的变更都必须包含：
- 改了什么（高层结果）。
- 为什么改（解决的问题/风险）。
- 如何验证（命令、测试、人工检查）。
- 已知限制与后续计划。

## 11. PR 必需验收证据
PR 描述至少应包含：
- 变更摘要（前端/后端/共享层）。
- 验证命令与结果。
- UI 变更截图或交互记录。
- 后端契约变更示例。
- 风险评估与回滚说明。

## 12. 推荐命令基线
可随仓库演进调整，但基线应至少包含：

```bash
# 前端
npm --workspace @qcspec/web run lint
npm --workspace @qcspec/web run build

# 后端（示例，请替换为仓库实际命令）
# 运行 lint/type checks
# 运行 unit/integration tests
```

## 13. 执行与豁免
- 本标准是默认上线验收基线。
- 任何例外必须在 PR 中明确说明，并由指定评审/负责人批准。
- 重复违反必须有整改动作：方案、责任人、截止日期。
